1. HOW THE XAPP GETS METRICS (E2SM KPM SUBSCRIPTION)
================================================================================

1.1 Subscription Method:
-----------------------
- Uses E2SM KPM (Key Performance Metrics) subscription
- Subscribes to RAN Function ID = 2 (KPM_STATS_V3_0)
- Uses Format 3 indication messages (kpm_ind_msg_format_3_t)
- Reporting period: 100ms (period_ms = 100)
- Subscription callback: sm_cb_kpm()

1.2 Data Flow:
--------------
1. xApp connects to Near-RT RIC
2. Discovers available RAN functions (E2 nodes)
3. Subscribes to KPM RAN Function (ID=2) with 100ms period
4. Receives KPM indication messages via callback
5. Processes measurements in log_kpm_measurements()
6. Makes energy saving decisions based on SINR data

================================================================================
2. METRICS USED BY THE XAPP
================================================================================

2.1 PRIMARY METRICS (Used for Decision Making):
-----------------------------------------------

A) L3servingSINR3gpp_cell_{cellID}_UEID_{ueID}
   - Description: Serving cell SINR for specific UE in 3GPP format
   - Usage: Tracks UE connection quality to serving cell
   - Data Type: Real value (double)
   - Example: "L3servingSINR3gpp_cell_5_UEID_00002, sinr= 2.0000 [db]"

B) L3neighSINRListOf_UEID_{ueID}_of_Cell_{cellID}
   - Description: Neighbor cell SINR measurements for UE
   - Usage: Determines best target cell for handover
   - Data Type: Real value (double) + Integer (neighbor cell ID)
   - Example: "L3neighSINRListOf_UEID_00002, Neighbour=4, sinr= -12.0000 [db]"

2.2 SECONDARY METRICS (Logged but Not Used for Decisions):
----------------------------------------------------------

C) RRU.PrbTotDl / RRU.PrbTotUl
   - Description: Total Physical Resource Blocks (PRBs)
   - Usage: Resource utilization monitoring
   - Data Type: Integer

D) DRB.PdcpSduVolumeDL / DRB.PdcpSduVolumeUL
   - Description: PDCP SDU volume in DL/UL
   - Usage: Throughput monitoring
   - Data Type: Integer (kb)

E) DRB.RlcSduDelayDl
   - Description: RLC SDU delay in downlink
   - Usage: Latency monitoring
   - Data Type: Real value (μs)

F) DRB.UEThpDl / DRB.UEThpUl
   - Description: UE throughput in DL/UL
   - Usage: Throughput monitoring
   - Data Type: Real value (kbps)

================================================================================
3. XAPP DECISION LOGIC
================================================================================

3.1 Energy Saving Algorithm:
----------------------------
1. Monitor serving SINR for each UE in each cell
2. Collect neighbor SINR measurements for handover decisions
3. Identify cells with low utilization (≤1 UE) for shutdown
4. For cells marked for shutdown:
   - Find best target cell for each UE (highest SINR > -10 dB)
   - Trigger handover using E2SM RC (RAN Control)
   - Wait for handover completion or timeout (30s)
   - Switch off cell using E2SM RC control

3.2 Key Decision Parameters:
---------------------------
- MIN_SINR = -10 dB (minimum acceptable SINR)
- SHUTDOWN_TIMEOUT_SEC = 30 seconds
- MAX_NUM_OF_RIC_INDICATIONS = 5 (measurement samples)
- Cell shutdown criteria: ≤1 connected UE

4. CAN WE GET THESE METRICS FROM e2andLogging.txt FILES?
================================================================================

4.1 SERVING SINR METRICS:
-------------------------
✅ AVAILABLE: cu-cp-cell-*.txt files
   - Column: "L3 serving SINR 3gpp"
   - Period: 100ms (matches xApp requirement)
   - Format: Real value in dB
   - Example: "L3servingSINR3gpp_cell_5_UEID_00002" → "L3 serving SINR 3gpp" column

✅ AVAILABLE: du-cell-*.txt files  
   - Columns: "L1M.RS-SINR.Bin34" to "L1M.RS-SINR.Bin127"
   - Period: 100ms
   - Format: Integer bins (can be converted to dB)
   - SINR = (bin - 64) * 0.5 dB

4.2 NEIGHBOR SINR METRICS:
--------------------------
✅ AVAILABLE: cu-cp-cell-*.txt files
   - Columns: "L3 neigh SINR 3gpp 1" to "L3 neigh SINR 3gpp 8"
   - Period: 100ms
   - Format: Real values in dB
   - Neighbor cell IDs: "L3 neigh Id 1" to "L3 neigh Id 8"

4.3 RESOURCE UTILIZATION METRICS:
---------------------------------
✅ AVAILABLE: du-cell-*.txt files
   - Column: "RRU.PrbUsedDl" (matches RRU.PrbTotDl)
   - Period: 100ms
   - Format: Integer (PRB usage)

✅ AVAILABLE: cu-up-cell-*.txt files
   - Column: "DRB.PdcpSduVolumeDl_Filter.UEID (txBytes)"
   - Period: 100ms  
   - Format: Integer (matches DRB.PdcpSduVolumeDL)

4.4 THROUGHPUT METRICS:
-----------------------
✅ AVAILABLE: cu-up-cell-*.txt files
   - Column: "DRB.PdcpSduBitRateDl.UEID(pdcpThroughput)"
   - Period: 100ms
   - Format: Real value (matches DRB.UEThpDl)

4.5 DELAY METRICS:
------------------
✅ AVAILABLE: cu-up-cell-*.txt files
   - Column: "DRB.PdcpSduDelayDl.UEID (pdcpLatency)"
   - Period: 100ms
   - Format: Real value (matches DRB.RlcSduDelayDl)

6. ANSWER: CAN WE REPLACE E2SM KPM WITH e2andLogging.txt?
================================================================================

✅ YES, ALL METRICS ARE AVAILABLE!

6.1 Advantages of Using e2andLogging.txt:
------------------------------------------
- No need for E2SM KPM subscription setup
- No need for Near-RT RIC connection
- Simpler data processing (CSV format)
- All required metrics available at 100ms period
- Direct file access (no callback complexity)

6.2 Implementation Approach:
----------------------------
1. Read cu-cp-cell-*.txt files for SINR metrics
2. Read du-cell-*.txt files for resource utilization
3. Read cu-up-cell-*.txt files for throughput/delay
4. Parse CSV data every 100ms
5. Implement same decision logic as xApp
6. Send control commands via E2SM RC (if needed)

6.3 Data Processing Example:
----------------------------
```python
# Read serving SINR for UE 2 in cell 5
serving_sinr = parse_cu_cp_file("cu-cp-cell-5.txt", ue_id=2, column="L3 serving SINR 3gpp")

# Read neighbor SINRs for UE 2 in cell 5  
neighbor_sinrs = []
for i in range(1, 9):
    neigh_id = parse_cu_cp_file("cu-cp-cell-5.txt", ue_id=2, column=f"L3 neigh Id {i}")
    neigh_sinr = parse_cu_cp_file("cu-cp-cell-5.txt", ue_id=2, column=f"L3 neigh SINR 3gpp {i}")
    if neigh_id != 0:  # Valid neighbor
        neighbor_sinrs.append((neigh_id, neigh_sinr))

# Find best target cell (same logic as xApp)
best_target = max(neighbor_sinrs, key=lambda x: x[1] if x[1] > -10 else -999)
```
